### 통합 프로세스(United Process) 모델

UP는 Jacobson, Booch, Rumbaugh에 의해 1999년에 개발된 **객체지향** 소프트웨어 개발 방법론이다.



### UP의 특징

- UP는 **반복적(Iterative)**이고 **점진적(Incremental)**인 개발 방법론이다.
- **유스케이스(Use Case)**기반으로 한다 -> 객체지향적
- **아키텍처(Architecture) 중심**의 개발을 지향한다.
- **위험 관리**를 중시한다.



### UP의 개발 단계

<u>IECT(Inception, Elaboration, Construction, Transition)</u>

1. **Inception(도입, 준비, 인지) 단계**
2. **Elaboration(상세, 구체화) 단계**
3. **Construction(구축) 단계**
4. **Transition(이행, 전이) 단계**



### Inception(인지, 도입) 단계

- 도입은 **소프트웨어 개발에 관련된 사람들(고객, 사용자 등)** 과 **개발을 준비**하는 단계
- **요구사항 관련 작업** 의 비중이 크게 나타난다.
- **프로젝트 목표와 실현가능성** 에 대해 파악하고, **대략적인 비용 평가** 를 통해 **프로젝트 개발 여부를 결정** 하는 등 **전체 요구사항을 이해** 하는 데 중점을 둔다.

| 요구사항 | 분석 | 설계 | 구현 | 평가 |
| -------- | ---- | ---- | ---- | ---- |
| 1        | 2    | X    | X    | X    |



### Elaboration(상세, 구체화) 단계

- 상세는 대부분의 **요구사항** 들과 개발 시스템에 대한 **기본적인 아키텍처가 작성** 되는 단계이다
- **요구사항, 분석** 그리고 아키텍처 과정을 위한 **설계** 관련 작업의 비중이 크게 나타나며 시스템 구성에 관련된 **위험 요소를 식별**하고 이를 완화하는데 중점을 둔다.

| 요구사항 | 분석 | 설계 | 구현 | 평가 |
| -------- | ---- | ---- | ---- | ---- |
| 1        | 1    | 1    | X    | X    |



### Construction(구축) 단계

- 구축은 **설계를 기반** 으로 시스템을 구축하는 단계이다.
- 전반적으로 시스템을 **구현** 하고 **평가** 하는 작업의 비중이 크게 나타난다.

| 요구사항 | 분석 | 설계 | 구현 | 평가 |
| -------- | ---- | ---- | ---- | ---- |
| X        | X    | 2    | 1    | 1    |



### Transition(이행) 단계

- 이행은 **제품의 완성** 단계이다
- **설계, 구현** 그리고 **평가활동** 의 비중이 크게 나타난다. 그렇기 때문에 **시스템 개발을 완료하고 그에 따른 품질을 보장** 하여 **사용자에게 제품을 인도** 하는데 중점을 둔다.
- 완성된 시스템 및 산출물들을 사용자에게 인도하기 위하여 **사용자 환경 인수 테스트** 가 수행되고 **시스템에 대한 사용자 교육 및 훈련** 이 수행된다.

| -------- | ---- | ---- | ---- | ---- |
| X        | X    | 1    | 1    | 1    |
